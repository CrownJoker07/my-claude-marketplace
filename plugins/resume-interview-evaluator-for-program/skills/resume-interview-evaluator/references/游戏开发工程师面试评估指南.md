## 面试流程概览

| 阶段 | 时长 | 考察目标 | 核心内容 |
|------|------|---------|---------|
| 1. 自我介绍 | 2分钟 | 职业规划、求职动机 | 背景介绍+岗位认知 |
| 2. 项目经历 | 20分钟 | 技术深度、项目真实性 | 项目概述+技术深挖+问题解决 |
| 3. 基础能力 | 5分钟 | 计算机基础、编程能力 | 语言/数据结构/算法/设计模式 |
| 4. 非技术素质 | 2分钟 | 团队协作、抗压能力 | 冲突处理/压力应对/学习方法 |
| 5. 候选人提问 | 3分钟 | 岗位关注度 | 候选人主动提问 |
| **总计** | **32分钟** | | |

## 第一阶段：自我介绍（2分钟）

### 核心问题清单

- [ ] **请简单介绍一下自己**（1分钟）
  - 观察点：逻辑性、重点突出程度

- [ ] **为什么来面试这个岗位？**
  - 观察点：是否有清晰的职业规划，还是随大流

- [ ] **为什么选择我们公司/项目组？**
  - 观察点：是否了解公司业务，求职诚意

### 评估要点

| 维度 | 优秀表现（90-100） | 良好（70-89） | 需关注（<70） |
|------|------------------|--------------|--------------|
| **职业规划** | 有清晰短期+长期目标，与岗位匹配 | 有目标但不够具体 | 目标模糊或随大流 |
| **岗位认知** | 准确了解岗位职责和要求 | 基本了解但有偏差 | 对岗位理解错误 |
| **表达能力** | 逻辑清晰，重点突出，时间把控好 | 较清晰但略有冗余 | 过于冗长或过于简短 |

### 追问技巧

如果候选人回答过于简略，可以追问：
- "你未来的3年职业规划是什么？"
- "你认为自己最适合什么样的工作？"
- "这个岗位最吸引你的地方是什么？"

## 第二阶段：项目经历深挖（20分钟）

### 2.1 项目概述（3分钟）

- [ ] **请介绍你最熟悉的项目**
  - 项目背景、类型、规模

- [ ] **你在项目中担任什么角色？**
  - 主程/模块负责人/参与者

- [ ] **团队规模？开发周期？**
  - 验证项目复杂度和个人贡献

### 2.2 技术细节深挖（12分钟）

根据简历技术栈选择 **2-3个方向** 深入追问：

#### Unity方向问题

**必问题：**
1. **项目整体架构是如何设计的？使用了什么设计模式？**
   - 考察点：架构设计能力、代码组织能力

2. **有没有遇到性能问题？如何分析和优化的？**
   - 考察点：性能优化意识、工具使用能力
   - 追问：使用了什么Profiler工具？具体优化了多少帧率/内存？

3. **资源管理使用的是什么方案？(Resources/AB/Addressable) 为什么选择这个？**
   - 考察点：技术选型能力、对Unity资源系统的理解

4. **UniTask和Coroutine的区别是什么？项目中如何选择？**
   - 考察点：异步编程理解、现代Unity开发实践

#### 架构设计方向问题

**必问题：**
1. **简历中提到"XXX"，请具体说明是如何应用的？**（针对简历中可能存在夸大的技术点）
   - 考察点：验证技术描述真实性

2. **战斗系统/AI系统是如何设计的？如何应对需求变更？**
   - 考察点：系统设计能力、扩展性考虑

3. **如何保证代码的可扩展性和可维护性？**
   - 考察点：代码质量意识、工程化思维

#### 渲染/图形方向问题（如岗位需要）

1. **请简单描述Unity的渲染流程**
2. **LOD和视锥剔除是如何实现的？带来了多少性能提升？**
3. **Shader是如何编写的？顶点着色器和片元着色器分别做什么？**

### 2.3 问题与解决（5分钟）

- [ ] **项目中遇到的最大技术挑战是什么？**
  - 观察候选人如何定义"挑战"

- [ ] **如何解决的？如果重新设计会怎么做？**
  - 考察问题解决能力和反思能力

- [ ] **项目中有没有失败或遗憾的地方？**
  - 考察自我认知和诚实度

### 评估要点

| 维度 | 权重 | 优秀表现 | 评分 |
|------|------|---------|------|
| **技术深度** | 30% | 能讲清楚原理和实现细节，有优化意识 | /30 |
| **项目真实性** | 25% | 细节一致，经得起追问，不矛盾 | /25 |
| **问题解决** | 25% | 有具体案例，思路清晰，能举一反三 | /25 |
| **技术视野** | 20% | 了解技术选型的trade-off，不盲从 | /20 |
| **小计** | **100%** | | **/100** |

## 第三阶段：基础能力考察（5分钟）

> 注：项目经历中已覆盖的部分可跳过，补充考察薄弱环节

### 3.1 编程语言（C#）

- [ ] **值类型和引用类型的区别？**
  - 延伸：什么时候会发生装箱拆箱？如何避免？

- [ ] **什么是GC？如何避免GC Alloc？**
  - 延伸：对象池的原理是什么？

- [ ] **async/await的原理是什么？**
  - 延伸：和Coroutine的区别？

### 3.2 数据结构

- [ ] **列举常用的数据结构及其适用场景**
  - List/Dictionary/Queue/Stack/HashSet

- [ ] **游戏中常用什么数据结构管理游戏对象？**
  - 延伸：四叉树、八叉树、空间哈希的应用

### 3.3 算法

- [ ] **解释A*寻路算法的原理**
  - 延伸：和Dijkstra算法的区别？启发函数如何选择？

- [ ] **如何实现一个对象池？有什么好处？**
  - 考察点：内存管理、性能优化意识

### 3.4 设计模式

- [ ] **项目中使用了哪些设计模式？解决什么问题？**

- [ ] **单例模式的优缺点？有什么替代方案？**
  - 延伸：DI（依赖注入）是什么？

### 3.5 计算机图形学（可选，根据岗位）

- [ ] **什么是MVP矩阵？**
- [ ] **简述Unity的渲染管线**

## 第四阶段：非技术素质（2分钟）

### 核心问题

- [ ] **项目与他人合作有没有遇到过矛盾？如何处理？**
  - 考察点：团队协作能力、冲突处理能力

- [ ] **合理安排需求的情况下，没做完的需求会如何处理？**
  - 考察点：时间管理能力、抗压能力、沟通能力

- [ ] **工作过程中遇到不懂的问题会如何解决？假设工作日中也没解决，会怎么办？**
  - 考察点：学习能力、主动性、求助能力

### 评估要点

| 维度 | 优秀表现 | 需关注 |
|------|---------|--------|
| **协作能力** | 能妥协，会沟通，主动同步进度 | 过于固执或推卸责任 |
| **抗压能力** | 积极面对，主动求助，不逃避 | 被动等待或消极应对 |
| **学习能力** | 有系统学习方法，善于总结 | 无章法，纯试错，不总结 |

## 第五阶段：候选人提问（3分钟）

- [ ] **给候选人提问机会**
  - 观察点：是否关心团队、项目、技术成长
  - 警惕：只问薪资福利，不关心工作内容

### 常见问题类型

| 问题类型 | 评估 |
|---------|------|
| 团队技术栈/架构 | 关心技术环境 |
| 项目类型/挑战 | 关心工作内容 |
| 技术成长/导师 | 关心个人发展 |
| 薪资/加班/福利 | 只关心待遇（需谨慎） |

## 面试问题库

### Unity引擎方向

#### 生命周期与组件
- Unity生命周期函数的执行顺序是什么？
- Awake和Start的区别？实际项目中如何选择？
- Update、FixedUpdate、LateUpdate的区别？
- GetComponent的原理？如何优化频繁获取组件？

#### 资源管理
- Resources.Load的优缺点？为什么大型项目不推荐？
- AssetBundle的工作原理？
- Addressable系统的优势和核心概念？
- 资源热更新的方案有哪些？

#### 性能优化
- Unity Profiler的主要功能和使用方法？
- Draw Call是什么？如何减少Draw Call？
- 什么是GC Alloc？如何避免？
- Object Pool的对象池原理和实现？
- LOD（多细节层次）的实现原理？
- 遮挡剔除和视锥剔除的区别？

#### UI系统
- UGUI的渲染流程？
- Canvas的三种渲染模式？
- UI优化有哪些注意点？

#### 动画系统
- Animator和Animation的区别？
- 状态机的设计原则？
- 动画混合（Blend Tree）的原理？

### C#编程方向

- 值类型和引用类型的区别？
- ref和out的区别？
- 什么是装箱拆箱？如何避免？
- 委托（Delegate）和事件（Event）的区别？
- 什么是反射（Reflection）？
- 泛型的优势？
- Task和Thread的区别？

### 设计模式方向

- 单例模式的优缺点？在Unity中的实现？
- 观察者模式（Observer）的应用场景？
- 工厂模式的作用？
- 状态模式（State Pattern）在游戏中的应用？
- MVC、MVP、MVVM的区别？
- ECS架构的核心思想？

### 算法与数据结构方向

- 数组和链表的区别？使用场景？
- 栈和队列的应用？
- 哈希表的原理？
- A*寻路算法的原理？与Dijkstra的区别？
- 二叉树遍历的几种方式？
- 动态规划的基本思想？

### AI与寻路方向

- A*算法的启发函数如何选择？
- NavMesh的工作原理？
- 行为树的节点类型？
- Selector和Sequence的区别？
- 行为树和状态机的对比？

### 网络与多线程方向

- TCP和UDP的区别？游戏开发中如何选择？
- 什么是帧同步和状态同步？
- Unity中的多线程限制？
- UniTask的优势？

### 简历深挖问题

#### 验证项目真实性
- 项目中最大的技术挑战是什么？如何解决的？
- 团队有几个人？分工如何？
- 项目开发了多久？
- 有没有使用版本控制？如何解决代码冲突？

#### 技术深度追问
- 如果重新设计这个项目，你会做哪些改进？
- 项目的性能瓶颈在哪里？如何优化的？
- 如何保证代码质量？

### 场景设计问题

- 设计一个简单的技能系统，需要考虑哪些因素？
- 如何实现一个背包系统？
- 设计一个简单的任务系统。
- 如何实现伤害计算系统？

### 开放性问题

- 你最擅长的技术方向是什么？
- 最近在学习什么新技术？
- 你认为好的代码应该具备什么特点？
- 对加班的看法？
- 未来的职业规划？

## 评分标准与评估表

### 评分维度权重

| 评估维度 | 权重 | 得分范围 | 说明 |
|---------|------|---------|------|
| **技术能力** | 35% | 0-100 | Unity/C#/架构设计/性能优化 |
| **项目经验** | 25% | 0-100 | 项目深度/复杂度/个人贡献 |
| **算法基础** | 15% | 0-100 | 数据结构/算法/设计模式 |
| **团队协作** | 10% | 0-100 | 沟通/协作/冲突处理 |
| **发展潜力** | 10% | 0-100 | 学习能力/技术视野/成长曲线 |
| **文化匹配** | 5% | 0-100 | 价值观/工作态度/稳定性 |
| **总分** | **100%** | **0-100** | |

### 各维度评分细则

#### 1. 技术能力（35%）

**Unity/C#基础（15分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 13-15 | 深入理解引擎原理，能优化性能问题 |
| 良好 | 10-12 | 熟练使用API，理解常见设计模式 |
| 一般 | 7-9 | 基本功能能实现，但缺乏优化意识 |
| 较差 | <7 | API不熟练，基础概念模糊 |

**架构设计能力（10分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 9-10 | 能设计可扩展的模块，合理使用设计模式 |
| 良好 | 7-8 | 代码结构清晰，有一定解耦意识 |
| 一般 | 5-6 | 代码能跑但结构混乱 |
| 较差 | <5 | 完全不懂架构，代码堆砌 |

**问题解决能力（10分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 9-10 | 有具体案例，思路清晰，能举一反三 |
| 良好 | 7-8 | 有解决思路，但不够系统 |
| 一般 | 5-6 | 能描述问题但解决思路模糊 |
| 较差 | <5 | 无法描述遇到的技术问题 |

#### 2. 项目经验（25%）

**项目复杂度（10分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 9-10 | 完整游戏项目，涉及多个系统 |
| 良好 | 7-8 | 完整模块开发，有一定复杂度 |
| 一般 | 5-6 | 小功能或Demo级别 |
| 较差 | <5 | 无实质项目经验 |

**个人贡献（10分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 9-10 | 核心开发者，能讲清技术决策 |
| 良好 | 7-8 | 独立负责模块，有一定自主性 |
| 一般 | 5-6 | 参与开发但主要是执行 |
| 较差 | <5 | 边缘角色或无法验证 |

**项目真实性（5分）**
| 等级 | 分数 | 标准 |
|------|------|------|
| 优秀 | 5 | 有作品链接，细节经得起追问 |
| 良好 | 4 | 描述合理，无明显矛盾 |
| 一般 | 3 | 有疑点但不确定 |
| 较差 | <3 | 明显夸大或造假 |

### 推荐等级标准

| 等级 | 总分 | 说明 | 处理建议 |
|------|------|------|---------|
| **A级（强烈推荐）** | ≥85 | 各维度无明显短板，至少一项优秀 | 优先录用，可给高于预期的薪资 |
| **B级（推荐）** | 70-84 | 无明显硬伤，有培养潜力 | 正常录用，标准薪资 |
| **C级（谨慎考虑）** | 60-69 | 有明显短板但可弥补，或有风险点 | 可给二面机会，或降低预期录用 |
| **D级（不推荐）** | <60 | 多个维度不达标，或有诚信问题 | 不录用 |

### 特殊情况处理

| 情况 | 处理建议 |
|------|---------|
| 高分但存在风险点 | 要求提供代码作品或增加一轮技术面试 |
| 低分但潜力明显 | 可考虑给予实习机会，重点培养 |
| 技术强但沟通差 | 根据团队情况决定，独立工作者可接受 |
| 经验少但基础扎实 | 可录用，给予培养时间 |
